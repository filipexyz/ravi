# UI Artifacts

## Intent

UI artifacts make generated interface work durable, inspectable, reusable, and
collaborative.

Ravi UI MUST be centered on artifacts when the UI object has independent reuse,
review, lineage, preview, rollback, or debugging value. The artifact is the
stable object; renderers are only surface-specific materializations of that
object.

This capability defines the generic Ravi contract inspired by json-render's
catalog, spec, registry, and JSON Patch streaming model. Ravi MAY interoperate
with json-render implementations, but this spec defines Ravi's portable
artifact model rather than adopting any one frontend runtime as the authority.

## Core Model

Ravi UI is composed from four layers:

1. `ui.catalog`: the vocabulary of components, actions, schemas, slots, and
   generation rules that an agent or event presenter may use.
2. `ui.component`: a reusable component contract, with semantic id, props
   schema, slots, events, actions, fixtures, tests, and surface renderers.
3. `ui.spec`: a JSON tree that composes catalog components into a concrete UI.
4. `ui.render`: a debug or review snapshot of a spec rendered on a surface,
   including input data, patches, final spec, screenshot or textual snapshot,
   and trace metadata.

The minimum renderable UI unit is `ui.spec`. The minimum reusable design unit is
`ui.component`. The minimum governance unit for allowed generation is
`ui.catalog`.

## Artifact Kinds

- `ui.catalog`
  - Versioned component/action vocabulary for one domain or surface class.
  - SHOULD include component ids, action ids, JSON Schema or equivalent props
    schemas, slot declarations, constraints, examples, and prompt/generation
    guidance.
- `ui.component`
  - Reusable component contract such as `artifact.card`, `artifact.drawer`,
    `notification.toast`, `session.lifeline`, or `agent.voice-picker`.
  - MUST be versioned when its props schema, behavior, actions, visual contract,
    slots, or renderer compatibility change.
- `ui.spec`
  - Concrete UI composition. It may be generated by an agent, emitted by an
    event presenter, stored as a reusable template, or hand-authored as JSON.
  - SHOULD use a flat `root` plus `elements` shape unless a capability-specific
    schema explicitly requires another representation.
- `ui.render`
  - Rendered instance for review, QA, debugging, or artifact notification.
  - SHOULD be created for important previews, failures, approvals, and
    developer diagnostics. It MUST NOT be required for every runtime render.
- `ui.patch-stream`
  - Optional artifact for a preserved UI stream. Individual patches SHOULD
    normally live in event/trace storage and SHOULD NOT become standalone
    artifacts unless the stream itself needs audit, replay, or debugging.

## UI Spec Shape

The preferred Ravi UI spec shape is a portable flat tree:

```json
{
  "schema": "ravi.ui.spec/v1",
  "catalog": {
    "artifactId": "art_catalog_...",
    "version": "1.2.0"
  },
  "root": "artifact-notification",
  "elements": {
    "artifact-notification": {
      "type": "notification.toast",
      "version": "^1.0.0",
      "props": {
        "title": "Artifact completed",
        "tone": "success"
      },
      "children": ["artifact-card"]
    },
    "artifact-card": {
      "type": "artifact.card",
      "version": "^1.0.0",
      "props": {
        "artifactId": "art_...",
        "status": "completed"
      },
      "children": []
    }
  },
  "state": {
    "artifact": {
      "id": "art_...",
      "kind": "image",
      "status": "completed"
    }
  },
  "actions": {
    "open": {
      "operation": "artifacts.show",
      "params": {
        "id": "art_..."
      }
    }
  }
}
```

Rules:

- `schema` MUST identify the UI spec schema version.
- `catalog` SHOULD reference a catalog artifact or named registry release.
- `root` MUST point to an element id present in `elements`.
- `elements` MUST be an object keyed by stable element ids.
- Each element MUST declare `type`, `props`, and `children`.
- Element `type` MUST resolve through the active catalog.
- Element `version` SHOULD use a semver range when component compatibility is
  intended and an exact version when deterministic replay is required.
- Props MUST validate against the component props schema.
- Children MUST reference ids present in `elements`.
- Runtime state bindings MUST use structured references, not string-templated
  HTML, CSS, or JavaScript.

## Catalog Contract

A `ui.catalog` artifact MUST define the allowed generation vocabulary:

```json
{
  "schema": "ravi.ui.catalog/v1",
  "id": "ravi.dev.notifications",
  "version": "1.0.0",
  "components": {
    "notification.toast": {
      "propsSchema": "schemas/notification-toast.props.v1.json",
      "slots": ["default"],
      "description": "Transient notification shell for recent system events"
    },
    "artifact.card": {
      "propsSchema": "schemas/artifact-card.props.v1.json",
      "slots": [],
      "description": "Compact artifact summary with status and primary action"
    }
  },
  "actions": {
    "artifacts.show": {
      "paramsSchema": "schemas/artifacts-show.params.v1.json",
      "description": "Open artifact details"
    }
  }
}
```

Rules:

- Catalogs MUST define what can be generated. Agents MUST NOT invent component
  ids, action ids, prop names, or slot names outside the active catalog.
- Catalogs SHOULD be artifacts so they can be versioned, reviewed, diffed, and
  linked to generated specs and renders.
- Catalog prompt/generation guidance MUST be derived from reviewed catalog
  metadata. It MUST NOT depend on hidden frontend implementation details.
- Catalogs MAY be domain-specific, surface-specific, or shared across Ravi.
  Shared catalogs SHOULD prefer semantic components over surface-native widgets.

## Component Contract

A `ui.component` artifact MUST document:

- stable `id`, such as `artifact.card`;
- semver `version`;
- `propsSchema`;
- named `slots`;
- supported `actions` and emitted `events`;
- supported `surfaces`, such as `wa-overlay`, `web-os`, `desktop`, `mobile`,
  `terminal`, `image`, `pdf`, or `email`;
- renderer references, implementation package references, or reviewed renderer
  source assets per surface;
- fixtures and sample state;
- accessibility, density, and interaction requirements;
- visual snapshots or preview artifacts when available;
- migration notes when a version breaks compatibility.

Agents MAY propose new component artifacts, but a component artifact SHOULD NOT
become active in a shared catalog until reviewed and validated.

## Renderer Boundary

Renderers translate `ui.spec` into native UI for a surface.

- A renderer MUST validate the spec against the referenced catalog before
  rendering.
- A renderer MUST fail closed on unknown components, invalid props, invalid
  action ids, invalid children, or unresolved required renderer implementations.
- A renderer MUST own DOM, CSS, accessibility, animation, input handling, and
  surface-specific security constraints.
- A generated spec MUST NOT include executable JavaScript, HTML strings, raw CSS,
  class names, Tailwind classes, or arbitrary remote bundles.
- A `ui.component` MAY include renderer JS/CSS/TSX source assets when those
  assets are part of the versioned component implementation, not generated
  per-render spec content.
- Surfaces that execute component-carried JS/CSS MUST do so through a reviewed
  renderer loader or a sandboxed runtime. Host-page direct injection is
  forbidden.
- Surface renderers MAY use React, Vue, Svelte, native views, Ink, PDF, image
  generation, or another implementation internally. That choice is not part of
  the portable spec contract.

## Streaming Contract

Ravi UI streams SHOULD use JSON Patch-style operations over a `ui.spec` object:

```json
{"op":"add","path":"/root","value":"artifact-notification"}
{"op":"add","path":"/elements/artifact-notification","value":{"type":"notification.toast","props":{"tone":"success"},"children":["artifact-card"]}}
{"op":"add","path":"/elements/artifact-card","value":{"type":"artifact.card","props":{"artifactId":"art_..."},"children":[]}}
```

Rules:

- Patch paths MUST be JSON Pointer paths into the `ui.spec`.
- Patch operations SHOULD follow RFC 6902 semantics.
- Stream consumers MUST validate the accumulated spec before exposing actions.
- Runtime UI streams SHOULD stay in event or trace storage by default.
- A complete patch stream SHOULD become a `ui.patch-stream` artifact only when
  replay, audit, approval, or debugging value justifies persistence.
- A terminal `ui.render` artifact MAY snapshot the accumulated spec and rendered
  surface state for debugging.

## Artifact Relationships

UI artifacts SHOULD link to each other through artifact attachments or lineage:

```text
ui.catalog
  -> contains ui.component references
ui.spec
  -> generated_from ui.catalog
  -> uses ui.component versions
  -> derived_from domain event, app operation, or source artifact
ui.render
  -> renders ui.spec
  -> observes event trace / patch stream
  -> may attach screenshot, HTML-free snapshot, or text snapshot
```

When a UI spec is generated from a domain artifact event, lineage SHOULD preserve:

- source artifact id and version;
- source event topic and event id when available;
- producing session, agent, channel, and message context when available;
- catalog artifact/version;
- component ids and resolved versions;
- renderer surface and renderer version.

## Artifact Content Layout

UI artifact versions SHOULD use a predictable package layout so producers,
renderers, reviewers, and agents can inspect them without provider-specific
knowledge.

Recommended `ui.catalog` package:

```text
_ravi/ui.catalog.json
schemas/
examples/
guidance.md
```

Recommended `ui.component` package:

```text
_ravi/ui.component.json       # required
src/component.tsx             # optional, when source is part of the component
src/index.ts                  # optional, package exports
styles/component.css          # optional, only when CSS is part of the component
schemas/props.schema.json     # optional, when propsSchema is not inline
fixtures/default.json         # recommended for preview/testing
previews/                     # optional screenshots or rendered snapshots
renderers.json                # optional, when renderer refs are not inline
migrations.md                 # required only for breaking changes
```

A minimal component artifact MAY be just `_ravi/ui.component.json` with inline
`propsSchema`, `fixtures`, `surfaces`, and `renderers`. Extra files exist only
when they carry reusable source, styles, previews, examples, or migration
history that should be versioned independently.

Recommended `ui.spec` package:

```text
_ravi/ui.spec.json
state.json
patches.jsonl
```

Recommended `ui.render` package:

```text
_ravi/ui.render.json
final-spec.json
input-state.json
surface-snapshot.*
trace.json
```

Rules:

- `_ravi/*.json` manifests MUST identify schema, kind, artifact id when known,
  version, created source, and lineage references.
- Schema and fixture paths MUST be Ravi-relative package paths.
- Renderer references MUST identify reviewed implementation ids, packages, or
  reviewed source assets. They MUST NOT embed arbitrary executable source in a
  generated `ui.spec`.
- Component package source MAY use TSX for web-capable surfaces when the source
  itself is part of the reusable component. TSX is source, not a portable
  runtime contract; the portable contract remains the manifest, props schema,
  fixtures, actions, slots, and renderer references.
- Component packages MAY include CSS, CSS modules, design-token references, and
  static assets only when they are part of the reusable component contract.
  These files MUST be versioned as artifact package assets when present.
- A surface MUST NOT inject arbitrary packaged JS/CSS into a host app directly.
  It MUST render through a reviewed local renderer, a build step that emits a
  trusted surface bundle, or a sandboxed preview runtime with explicit
  permissions.
- The shadcn-like model for Ravi is source-first and copyable: a component
  artifact can carry TSX/CSS/source assets for collaboration and versioning,
  while catalogs decide which reviewed component versions are allowed for
  generation.
- Preview assets SHOULD be attached as artifact version assets when available.
- Preserved `patches.jsonl` MUST use the streaming contract defined above.

## Ravi Apps Integration

Ravi App UI descriptors remain semantic app manifests. They MUST NOT embed raw
frontend code. When an app needs reusable or generative UI, it SHOULD reference
UI artifacts instead of expanding large frontend trees inline.

Recommended app UI reference shape:

```json
{
  "interfaces": {
    "ui": {
      "views": [
        {
          "id": "artifact-feed",
          "type": "timeline",
          "uiArtifact": {
            "kind": "ui.spec",
            "artifactId": "art_ui_spec_...",
            "version": 3
          },
          "query": {
            "operation": "artifacts.list"
          },
          "refreshOn": ["ravi.artifacts.completed"]
        }
      ]
    }
  }
}
```

The app manifest selects operations, routes, event freshness, and a referenced
UI artifact. The artifact registry owns reusable components, schemas, previews,
and renderer compatibility.

## Notification Pattern

Developer notifications SHOULD be modeled as UI artifacts:

1. A domain event occurs, such as an artifact completion.
2. A presenter maps the event into a `ui.spec` or patch stream using an approved
   catalog.
3. The active surface renderer shows a compact notification.
4. Expanding the notification renders artifact details, lineage, events, and
   debug metadata.
5. Important previews or failures create `ui.render` artifacts for later
   inspection.

This is the pilot pattern for artifact notifications and SHOULD generalize later
to sessions, tasks, voice, routes, permissions, and other Ravi events.

## Boundaries

- Do not treat every DOM node or transient runtime patch as an artifact.
- Do not persist every render. Persist only reusable components/specs/catalogs
  and renders that have review, debug, replay, or audit value.
- Do not let generated UI bypass Ravi operations or permission checks.
- Do not let component artifacts depend on commercial hosting or Pages product
  rules inside the open-source `ravi.bot` repository.
- Do not make `artifact.kind` determine publishability or renderer behavior.
  Renderer compatibility belongs to UI metadata and versions.
- Do not allow generated specs to introduce executable code.

## Validation

- `ravi specs get artifacts/ui --mode rules --json` MUST return this contract.
- A `ui.spec` MUST validate root/elements/children and catalog references before
  rendering.
- Component props MUST validate against the component props schema.
- App manifests that reference UI artifacts SHOULD validate that referenced
  artifacts exist and have compatible kind/schema when the artifact registry is
  available.
- Renderers SHOULD produce useful errors when a catalog, component, action,
  renderer, or props schema is missing.

## Known Failure Modes

- Treating generated UI as frontend code creates unsafe, unreviewed, and
  non-portable screens.
- Persisting every runtime patch as a standalone artifact floods the ledger.
- Letting agents invent component ids makes previews non-deterministic.
- Keeping UI specs outside artifacts loses lineage, approvals, fixtures, and
  rollback.
- Coupling specs to one frontend framework prevents reuse across the overlay,
  Web OS, mobile, terminal, PDF, image, and email surfaces.
