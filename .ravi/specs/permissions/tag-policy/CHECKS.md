# Tag-Driven Permission Policy Checks

These are target post-implementation checks for the `permissions policies`
surface. Until that CLI exists, run the SQL audits and inspect specs/code
manually.

## Unit Tests

- A non-policy tag such as `domain.core` attached to a contact does not create
  relations.
- A `policy.profile.trusted-dev` tag attached to a contact materializes
  `contact:<id> member role:trusted-dev` when the matching policy is enabled.
- The same policy defaults the generated relation to `grant_mode=temporary`
  with `expires_at` set.
- Removing the tag and reconciling revokes only the policy-owned generated
  relation.
- If an identical tuple exists with `source=manual`, policy materialization
  skips the row and records provenance without overwriting the manual source.
- A policy that emits `delegate_admin` is rejected during validation.
- A policy that emits `admin system:*` is rejected during validation.
- A policy using prefix tag matching must expose the concrete matched tag in
  explain output.
- A policy consuming an auto-generated `policy.*` tag is rejected unless the
  policy explicitly allows that tag source or auto-tagging rule id.
- A policy that emits `contact:c1 member role:owner` is rejected when
  `role:owner` closure contains `admin system:*`.
- A policy that emits membership into a role with `use tool:*`,
  `execute executable:*`, or `write_contacts system:*` is rejected unless the
  rule is break-glass approved with max TTL.
- A role change that adds forbidden output to a policy-managed role revokes or
  suspends existing policy-owned memberships before the next authority check.
- A tag detach triggers immediate reconciliation and invalidates runtime
  contexts whose effective capabilities depended on the policy-owned grant.

## Integration Tests

- Attach `policy.allow.tool.bash` to a chat and apply a policy that emits
  `chat:<id> delegate_use tool:Bash`; assert a resolved contact in that chat
  can receive Bash only when the executor agent also has `use tool:Bash`.
- Assert the same chat override does not allow Bash in another chat.
- Assert an agent-level `delegate_use tool:Bash` generated by policy still
  requires a surface grant or surface override.
- Attach `policy.profile.ops-apps` to a contact and assert profile expansion
  can expose `use app:khal-tasks` and `execute app:khal-tasks` only through the
  role grants.
- Assert `apps list` shows only apps visible after policy materialization and
  hides them after tag removal plus reconcile.

## CLI Smoke

```bash
ravi permissions policies validate --json
ravi permissions policies dry-run --json
ravi permissions policies explain --target contact:<contact-id> --json
```

Expected:

- validation lists invalid forbidden outputs;
- dry-run is read-only;
- explain includes tag slug, asset, rule id, emitted relation, lifetime, and
  whether the relation is active.
- explain includes materialization ledger status and conflict source when the
  desired tuple already exists from another source.

## SQL Audits

Policy-generated grants:

```sql
select subject_type, subject_id, relation, object_type, object_id,
       source, grant_mode, expires_at, revoked_at, reason, issued_by
from relations
where source like 'policy:%'
order by source, subject_type, subject_id, relation, object_type, object_id;
```

Policy materialization ledger:

```sql
select policy_id, policy_version, selector_asset_type, selector_asset_id,
       tag_slug, tag_binding_source, subject_type, subject_id, relation,
       object_type, object_id, status, relation_id, conflict_source,
       expires_at, revoked_at
from permission_policy_materializations
order by policy_id, selector_asset_type, selector_asset_id, relation;
```

Forbidden generated grants:

```sql
select *
from relations
where source like 'policy:%'
  and (
    relation = 'delegate_admin'
    or (relation = 'admin' and object_type = 'system' and object_id = '*')
    or (relation = 'use' and object_type = 'tool' and object_id = '*')
    or (relation = 'execute' and object_type = 'executable' and object_id = '*')
    or (relation = 'execute' and object_type = 'group' and object_id = '*')
    or (relation = 'write_contacts' and object_type = 'system' and object_id = '*')
    or (relation = 'read_tagged_contacts' and object_type = 'system' and object_id = '*')
  );
```

Broad legacy contact tag scopes:

```sql
select subject_type, subject_id, relation, object_type, object_id, source
from relations
where relation = 'read_tagged_contacts'
  and object_type = 'system'
  and object_id = '*'
  and revoked_at is null
order by subject_type, subject_id;
```

Policy tags by asset:

```sql
select b.asset_type, b.asset_id, d.slug, b.source, b.created_at
from tag_bindings b
join tag_definitions d on d.id = b.tag_id
where d.slug like 'policy.%'
order by d.slug, b.asset_type, b.asset_id;
```
